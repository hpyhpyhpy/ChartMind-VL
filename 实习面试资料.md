# 实习面试资料

## 阶段 1：本地训练前脚手架

### 做了什么

这一阶段先把不需要 GPU 的项目基础打好，避免一上来就在云端边训练边改结构。

主要完成了这些事：

- 建了项目的 Python 包结构：`src/chartvqa/`。
- 写了第一版依赖文件：`requirements.txt`。
- 写了第一版训练配置：`configs/qwen25vl_chartqa.yaml`，里面包含模型、数据集、LoRA、训练、评估和输出目录。
- 写了配置加载函数 `load_config`，保证后续训练、评估和 Demo 都从同一份配置读参数。
- 写了 ChartQA 样本规范化逻辑，把原始数据里的 `image`、`query`、`label` 等字段统一成 `image`、`question`、`answer`、`question_type`。
- 写了 Qwen2.5-VL 的 chat messages 格式化逻辑，把一条图表问答样本转成 system、user、assistant 三轮消息。
- 写了 AutoDL 环境检查脚本，用来在远程机器上检查 Python、CUDA、GPU 和依赖安装。
- 写了本地单元测试，确认配置、数据规范化和 prompt 格式化能在不加载模型、不下载数据、不占用 GPU 的情况下跑通。

### 解决了什么问题

这一阶段解决的是“训练前不确定性”的问题。

如果直接上云端跑训练，很容易把错误混在一起：可能是配置问题、字段问题、prompt 格式问题、模型加载问题，也可能是显存问题。现在先把配置、数据字段和 prompt 这些纯代码部分在本地测掉，后面云端失败时就更容易判断问题是不是来自模型加载、依赖版本或显存。

### 成果

- 本地已经有了可测试的训练前链路。
- 后续训练脚本可以复用同一套配置、数据样本结构和 prompt 格式。
- Claude Code 远程执行时有了明确的环境检查入口。
- 项目开始具备“工程化微调项目”的样子，而不是只停留在 notebook 复现。

### 面试问题与答案

**问题 1：为什么你没有一开始就直接写训练脚本，而是先做配置、数据和 prompt 脚手架？**

回答：因为多模态微调项目里，训练失败不一定是模型问题，很多时候是数据字段、prompt 格式、配置路径或者依赖环境出了问题。我先把不需要 GPU 的部分在本地拆出来测试，能提前排除一批低级但很耗时的问题。这样到了 AutoDL 上跑 smoke training 时，如果失败，就更容易定位到模型加载、显存、量化配置或者训练参数，而不是从头猜。

**问题 2：ChartQA 原始样本为什么要先规范化成统一的 `ChartQASample`？**

回答：因为不同数据集或者参考 notebook 里的字段命名可能不一样，比如有的叫 `query` 和 `label`，有的叫 `question` 和 `answer`。如果训练、评估、Demo 每个地方都直接读原始字段，后面换数据集或扩展业务报表样例会很乱。我先定义统一结构，让下游只关心 `image`、`question`、`answer`，这样数据源变化时主要改数据适配层，不影响训练和推理逻辑。

**问题 3：Qwen2.5-VL 的 prompt 为什么要做成 messages，而不是简单拼接字符串？**

回答：因为 Qwen2.5-VL 是多模态 chat 模型，输入里既有图片也有文本问题。messages 结构可以清楚表达 system 指令、user 的 image/text 内容和 assistant 答案，后续再交给 processor 的 chat template 处理。这样更贴近模型官方使用方式，也方便训练时对 assistant 部分做 label，避免把用户问题也当作模型要学习输出的内容。

**问题 4：这个阶段为什么不下载完整 ChartQA 或加载 Qwen2.5-VL？**

回答：这是有意控制范围。完整数据集下载和 7B 多模态模型加载都依赖网络、磁盘、显存和 CUDA 环境，适合放到 AutoDL 上验证。本地阶段只验证纯逻辑，保证代码结构正确。这样本地迭代快，远程验证也更聚焦。

**问题 5：如果面试官问这个项目和普通 notebook 复现有什么区别，可以怎么说？**

回答：普通 notebook 复现往往把数据处理、prompt、模型加载、训练和评估写在一个文件里，能跑但不好维护。我这个项目把它拆成配置层、数据适配层、prompt 层、后续训练评估层，并且用测试覆盖不依赖 GPU 的部分。这样更接近真实工程项目：可以复现实验，可以替换数据，可以分析 badcase，也能把训练和 Demo 分开维护。

## 阶段 2：远程 Smoke Training 入口

### 做了什么

这一阶段继续基于 Kaggle 上的 Qwen2.5-VL + ChartQA 微调 notebook，把里面的训练关键步骤整理成可以在 AutoDL 上直接运行的项目脚本。

主要完成了这些事：

- 写了 `src/chartvqa/modeling.py`，专门负责模型相关逻辑。
- 把 4-bit BitsAndBytes 量化参数整理成配置构造函数，使用 NF4、double quantization 和 bfloat16 compute。
- 把 LoRA 参数整理成配置构造函数，第一版使用 `r=8`、`alpha=16`、`dropout=0.1`，target modules 是 `q_proj` 和 `v_proj`。
- 写了 Qwen2.5-VL model 和 processor 的加载入口，训练模式下会关闭 cache，并启用 gradient checkpointing。
- 写了 `src/chartvqa/training.py`，负责 TRL `SFTConfig` 参数、数据格式化、VLM collator 和 `SFTTrainer` 组装。
- 写了 `scripts/train_lora.py`，作为 Claude Code 在 AutoDL 上运行 smoke training 的标准命令入口。
- 给训练入口加了 `--max-steps` 参数，方便先跑 1 step 验证链路。
- 本地新增测试，只验证参数构造、数据格式化和命令行参数，不加载大模型、不占用 GPU。

### 解决了什么问题

这一阶段解决的是“云端到底运行什么命令”的问题。

如果没有标准训练入口，远程训练很容易变成在 notebook 里手动改 cell，失败后也不好复现。现在训练入口固定为：

```bash
python scripts/train_lora.py --config configs/qwen25vl_chartqa.yaml --max-steps 1 --skip-initial-eval
```

这样 Claude Code 在 AutoDL 上只需要执行固定命令，并把日志反馈回来。后面如果失败，我们能根据日志判断是模型下载、依赖版本、显存、数据集还是 collator 的问题。

### 成果

- 项目已经具备远程 1-step smoke training 的脚本入口。
- LoRA 和 4-bit 量化参数从 notebook 迁移到了工程化配置。
- 本地测试能验证训练参数链路，不需要 GPU。
- 下一步可以交给 Claude Code 在 AutoDL 上真实加载模型并跑 1 step。

### 面试问题与答案

**问题 1：为什么第一版 LoRA 只选择 `q_proj` 和 `v_proj`？**

回答：这是参考 Qwen2.5-VL + ChartQA notebook 的保守配置，也是很多 LoRA 微调里常见的低成本选择。`q_proj` 和 `v_proj` 属于 attention 里的关键投影层，调整它们可以影响模型对输入信息的关注方式，同时可训练参数量比较少，适合单卡 4090 做第一版 smoke test。等链路跑通后，再考虑扩大到更多模块或提高 rank。

**问题 2：为什么要用 4-bit QLoRA，而不是直接全量微调？**

回答：Qwen2.5-VL-7B 这种多模态 7B 模型全量微调显存成本很高，单卡 4090 24GB 不适合直接全量训练。4-bit QLoRA 通过量化冻结基础模型，再只训练 LoRA adapter，可以大幅降低显存占用，同时保留一定任务适配能力。这很适合项目第一阶段验证数据、prompt 和训练流程。

**问题 3：`--max-steps 1` 的 smoke test 有什么意义？它又不能代表最终效果。**

回答：`--max-steps 1` 不是为了看效果，而是为了验证工程链路：模型能不能下载和加载，数据能不能读，prompt 和 collator 能不能喂给 processor，LoRA 能不能注入，反向传播能不能跑，adapter 能不能保存。只有这一步成功，后面扩大 steps、数据比例和 epoch 才有意义。

**问题 4：为什么训练时要关闭 `use_cache` 并启用 gradient checkpointing？**

回答：训练时 `use_cache` 主要服务推理加速，但会占额外显存，也可能和 gradient checkpointing 不兼容。gradient checkpointing 会用重新计算换显存，适合单卡 4090 这种显存有限的环境。这个项目第一阶段更看重能稳定跑通，所以优先省显存。

**问题 5：VLM 的 collator 和普通文本模型有什么不同？**

回答：普通文本模型通常只处理 token ids 和 attention mask。VLM collator 还要处理图片输入，要把每条样本里的 chat messages 转成文本模板，同时提取图片传给 processor。最后还要构造 labels，并把 padding token mask 成 `-100`，避免模型学习 padding。这个步骤如果写错，训练可能能启动但 loss 和学习目标都是错的。

**问题 6：如果远程 smoke training 失败，你会怎么排查？**

回答：我会先看失败阶段。如果模型还没加载就失败，优先查依赖版本、模型下载、HuggingFace 访问和显存。 如果数据加载失败，查 ChartQA split 和字段。 如果进入 collator 后失败，查 messages 结构和 processor 输入。 如果反向传播 OOM，先调低 `max_pixels`、batch size、gradient accumulation 或跳过初始 eval。这样按阶段排查比只看最后一行报错更可靠。

## 阶段 3：Base vs LoRA 评估入口

### 做了什么

这一阶段在 smoke test 已经跑通之后，补上了评估入口，用来对比 Qwen2.5-VL base model 和 LoRA adapter 的效果。

主要完成了这些事：

- 写了 `src/chartvqa/evaluation.py`，实现 Exact Match、Token F1 和 Numeric Accuracy。
- 写了 `src/chartvqa/inference.py`，封装单张图表图片 + 问题的推理流程。
- 写了 `scripts/run_eval.py`，支持评估 `base`、`lora` 或 `both`。
- 评估脚本会输出逐样本 CSV：`reports/eval_results.csv`。
- 评估脚本会输出汇总 JSON：`reports/eval_summary.json`。
- 本地新增测试，验证评估指标和脚本参数解析，不加载大模型、不占用 GPU。

### 解决了什么问题

这一阶段解决的是“怎么证明 LoRA 有没有帮助”的问题。

只跑通训练还不够，因为训练能跑不代表模型效果更好。现在有了统一评估脚本，就可以在同一批 ChartQA test 样本上分别跑 base model 和 LoRA adapter，然后用相同指标对比。这样后续写 README、实验记录、简历和面试讲解时，都有数据支撑，而不是只展示 Demo 截图。

### 成果

- 项目从“能训练”推进到“能评估”。
- 评估结果可以直接支撑 badcase 分析。
- 后续 Gradio Demo 可以复用同一个推理封装。
- 远端第一轮建议先跑 20 条小样本，验证输出文件和推理流程，再扩大样本数。

远端建议命令：

```bash
python scripts/run_eval.py --config configs/qwen25vl_chartqa.yaml --mode both --adapter outputs/qwen25vl-chartqa-smoke --max-samples 20
```

### 面试问题与答案

**问题 1：为什么要同时做 Exact Match、Token F1 和 Numeric Accuracy？**

回答：ChartQA 里很多答案是短文本或数字。Exact Match 可以判断答案是否完全一致，但它太严格；Token F1 能给部分匹配一些分数，比如模型答出了关键实体但格式不完全一样；Numeric Accuracy 更关注图表问答里最关键的数值是否对。三者结合比只看一个指标更稳。

**问题 2：为什么 Numeric Accuracy 对图表问答很重要？**

回答：图表问答的核心经常是读数、比较和趋势判断。如果模型回答的文字看起来通顺，但数字错了，在业务场景里就不可用。比如财务报表、销售看板、运营指标这些场景，数值错误可能直接影响决策，所以必须单独评估数值是否匹配。

**问题 3：为什么要先做小样本评估，而不是直接跑完整 test set？**

回答：第一次评估的目标是验证推理、adapter 加载、指标计算和文件输出整条链路。完整 test set 会消耗较多时间和 GPU 费用，而且如果脚本有 bug，会浪费资源。先跑 20 条小样本能快速暴露路径、显存、依赖、输出格式等问题，确认稳定后再扩大。

**问题 4：base vs LoRA 对比要注意什么，才算公平？**

回答：要保证两边用同一批样本、同一套 prompt、同样的 max_new_tokens、同样的指标和后处理。否则差异可能来自评估设置，而不是 LoRA 本身。这个项目把推理和指标统一封装，就是为了让 base 和 LoRA 的对比尽量只差 adapter。

**问题 5：如果 LoRA 指标没有明显超过 base，怎么解释？**

回答：这不一定说明项目失败。当前 smoke adapter 只训练了很少步数，主要验证链路，不是追求最终效果。指标不提升时，可以从训练步数、数据比例、LoRA rank、学习率、prompt 格式和 badcase 类型继续分析。工程价值在于已经有了可重复训练和评估的闭环，后续才能系统优化。

## 阶段 4：正式小规模 LoRA 训练配置

### 做了什么

这一阶段根据远端评估 smoke test 的结果，准备了一份正式小规模训练配置。

主要完成了这些事：

- 新增 `configs/qwen25vl_chartqa_lora_1epoch.yaml`。
- 训练数据仍然先用 `train[:1%]`，控制成本和风险。
- 训练轮数设置为 `num_train_epochs: 1`。
- 把 `max_steps` 设置为 `-1`，避免继续使用 smoke test 的 20 steps 截断。
- 输出目录改为 `outputs/qwen25vl-chartqa-lora-1epoch`，避免覆盖 smoke adapter。
- 新增测试，确认正式训练配置会构造出 1 epoch 的 SFT 参数。
- 更新给 Claude Code 的交接文档，要求远端训练后再做 100 条样本评估。

### 解决了什么问题

上一阶段 base 和 LoRA 的 20 条评估指标完全一样。这不是代码失败，而是因为 LoRA 只训练了 20 steps，adapter 几乎还没学到东西。

这一阶段解决的是“让 LoRA 真正开始学习”的问题。我们没有直接扩大到完整数据集，而是先选择 `train[:1%]` 的完整 1 epoch。这样比 smoke test 更有学习信号，又不会一下子把训练成本和排错复杂度拉太高。

### 成果

- 项目进入从“链路验证”到“效果验证”的阶段。
- 正式 adapter 和 smoke adapter 分目录保存，方便对比和回退。
- 下一轮评估将使用 `outputs/qwen25vl-chartqa-lora-1epoch`，并跑 100 条 test 样本。

### 面试问题与答案

**问题 1：为什么不是直接用全量 ChartQA 训练？**

回答：项目第一阶段优先保证可控闭环。全量训练会增加训练时间、显存风险和排错成本。如果 1% 数据完整 1 epoch 都不能带来稳定结果，直接全量训练也很难判断问题在哪里。先小规模受控实验，可以更快定位学习率、LoRA rank、prompt 和数据处理是否合理。

**问题 2：为什么 `max_steps` 要从 20 改成 `-1`？**

回答：20 steps 是 smoke test，只验证训练链路能不能跑通。正式 1 epoch 训练要让 Trainer 按数据集和 epoch 自己计算训练步数，所以用 `max_steps: -1` 取消手动截断。否则配置看起来是 1 epoch，实际还是只训练 20 steps，容易误判实验结果。

**问题 3：为什么正式训练输出目录要和 smoke adapter 分开？**

回答：因为 smoke adapter 是链路验证产物，正式 adapter 是效果验证产物。如果都写到同一个目录，会覆盖历史结果，后续无法回溯，也不方便比较。工程实验里每组配置应该有独立输出，保证可复现和可解释。

**问题 4：如果 1 epoch 后 LoRA 仍然没有提升，下一步怎么做？**

回答：先看 badcase 和训练日志，不急着盲目扩大训练。可能原因包括数据量太小、学习率不合适、LoRA target modules 太少、prompt 格式不够稳定、评估样本太少，或者 base 本身已经很强。下一步可以对比 `r=16`、增加数据比例到 3%、调整学习率，或者按问题类型分析哪些样本有改善。

## 阶段 5：1 Epoch 训练结果与 Badcase 分析入口

### 做了什么

这一阶段 Claude Code 在远端完成了 ChartQA `train[:1%]` 的完整 1 epoch LoRA 训练，Codex 侧把结果沉淀成实验记录，并补了 badcase 分析工具。

主要完成了这些事：

- 记录 1 epoch 训练结果：36 steps，耗时 3 分 02 秒，adapter 约 4.9 MB。
- 记录 25 条样本评估结果：LoRA 在 EM、F1、Numeric Accuracy 三项指标上都超过 Base。
- 新增 `reports/experiments.md`，把 smoke training 和 1 epoch 训练整理成实验记录。
- 新增 `src/chartvqa/badcase.py`，用于对齐 base 和 LoRA 的逐样本预测。
- 新增 `scripts/analyze_badcases.py`，后续拿到远端 CSV 后可以生成 `reports/badcase_analysis.md`。
- 新增测试，保证 badcase 分类逻辑能识别 LoRA 改进、LoRA 退化、两者都对和两者都错。

### 解决了什么问题

这一阶段解决的是“有了指标提升之后，怎么解释提升从哪里来”的问题。

只说 LoRA 三项指标超过 Base 还不够，面试时更有说服力的是拿出具体样本：哪些题 base 错了但 LoRA 对了，哪些题两者都错，哪些题 LoRA 反而退化。badcase 分析入口就是为这个准备的。

### 成果

- LoRA 已经出现正向指标提升：
  - EM：0.20 -> 0.24
  - F1：0.28 -> 0.33
  - Numeric Accuracy：0.48 -> 0.52
- 项目从“训练能跑”推进到“训练有效果”。
- 后续可以基于逐样本 CSV 做案例分析和 Demo 样例筛选。

### 面试问题与答案

**问题 1：为什么 25 条样本上的提升还不能作为最终结论？**

回答：因为样本数太小，统计波动比较大。25 条样本能说明 LoRA 已经出现学习信号，但不能证明模型稳定优于 base。更严谨的做法是扩大评估集，比如 `test[:4%]` 或更多，再结合 badcase 分析看提升是否来自真实能力改进。

**问题 2：为什么 badcase 分析比只看总指标更重要？**

回答：总指标只能告诉我们平均表现变了，但不能告诉我们为什么变。图表问答里错误可能来自读错数值、理解错图例、比较关系错误或者输出格式不对。badcase 分析能把这些错误拆开，指导下一步是增大序列长度、调整 prompt、扩大数据，还是改评估口径。

**问题 3：LoRA 指标提升但 loss 没明显下降，怎么解释？**

回答：loss 和最终评估指标不一定同步，尤其当前训练数据只有 1%，步数只有 36 steps，而且 `max_seq_length=128` 限制了学习信号。评估指标出现提升说明 adapter 对部分样本已经有帮助，但 loss 曲线还不足以说明稳定收敛。后续需要更长训练、更大数据和更完整评估。

**问题 4：你会怎么选择 Demo 样例？**

回答：优先从 badcase 分析里挑 LoRA 改进的样本，也就是 base 答错但 LoRA 答对的样本。这类样例最能说明微调价值。然后再选两者都错的样本作为限制说明，体现项目不是只报喜，也能客观分析模型边界。

### 补充：Badcase 报告与脚本修复

远端生成了第一版 badcase 报告：

- LoRA 改进：1 条
- LoRA 退化：0 条
- 两者都对：12 条
- 两者都错：12 条

唯一的 LoRA 改进样本是一个 Yes/No 问题：base 回答了完整句子，LoRA 只回答 `Yes.`，更接近标准答案 `Yes`。这说明当前 LoRA 的一个可见收益是学到了更简洁的答案格式。

同时也发现 badcase 错误类型标注有 bug：已经正确的样本会被标成“完全不匹配”。Codex 已修复为：只要 EM=1 或 Numeric Accuracy=1，就先标记为“回答正确”。

**问题 5：为什么 LoRA 改进只有 1 条，还值得写进项目吗？**

回答：值得，但要诚实表达。当前训练只用了 1% 数据和 25 条评估样本，重点是证明训练和评估闭环已经跑通，并出现初步正向信号。这个结果不能夸大成显著提升，但可以说明项目具备继续扩大数据、调参和做 badcase 分析的基础。

**问题 6：发现 badcase 分类脚本有 bug，怎么处理才专业？**

回答：先复现问题，再补测试，然后修逻辑。这里的问题是正确样本没有优先判断，导致 EM=1 的行继续进入错误分类流程。修复方式是在 `describe_error_type` 一开始判断预测是否正确，正确就返回“回答正确”。这比手动改报告更可靠，因为后续重新生成报告也不会再错。

**问题 7：为什么分析脚本也要写测试？**

回答：实验分析脚本虽然不直接参与训练，但会影响我们怎么解释模型效果。如果分析脚本把正确样本标成错误，后面的 badcase 结论、Demo 样例选择和面试讲解都会被带偏。所以我把“正确预测应标记为回答正确”写成测试，保证以后重构或扩展 badcase 分类时不会复发。

### 面试问题与答案

**问题 1：为什么 Demo 要复用 `inference.py`，而不是在 `app.py` 里重新写推理逻辑？**

回答：因为评估和 Demo 应该走同一套推理路径。否则可能出现评估脚本是一种 prompt，Demo 又是另一种 prompt，最后指标和展示结果对不上。复用 `inference.py` 可以保证 base/LoRA 加载、prompt 格式和生成逻辑一致。

**问题 2：为什么 Demo 要支持 Base 和 LoRA 切换？**

回答：这个项目的核心不是单纯做一个问答应用，而是证明 LoRA 微调带来的变化。Base/LoRA 切换能直观看到微调前后的差异，也方便展示 LoRA 改进样本和失败样本。

**问题 3：为什么本地没有直接启动 Gradio？**

回答：本地没有安装 Gradio，也没有 GPU 和完整模型缓存。Codex 本地负责写代码和轻量测试，远端 AutoDL 负责真实模型加载和 Demo 启动。这样符合当前 Codex 与 Claude Code 的分工。

### 补充：远端 Demo 验证结果

Claude Code 已在远端完成 Demo 验证：

- 页面可以正常打开。
- Base 模式可以加载并回答。
- LoRA 模式可以加载并回答。
- Base 模型类型是 `Qwen2_5_VLForConditionalGeneration`。
- LoRA 模型类型是 `PeftModelForCausalLM`。

样本 24 中，Base 和 LoRA 都回答了 `Yes.`。这说明 Demo 加载没有问题，但由于当前只用了 1% 数据训练，LoRA 与 Base 的差异还比较微弱，随机测试不一定能看到明显差别。

**问题 4：Demo 中 Base 和 LoRA 回答一样，是否说明微调没用？**

回答：不一定。当前评估只有 25 条样本，LoRA 只在其中 1 条显示明显改进，而且改进主要是答案风格更简洁。Demo 随机测试很可能命中两者都对或两者都错的样本。判断微调是否有效应该看统一评估指标和 badcase 分析，而不是只看一次交互。
