# 实习面试资料

## 阶段 1：本地训练前脚手架

### 做了什么

这一阶段先把不需要 GPU 的项目基础打好，避免一上来就在云端边训练边改结构。

主要完成了这些事：

- 建了项目的 Python 包结构：`src/chartvqa/`。
- 写了第一版依赖文件：`requirements.txt`。
- 写了第一版训练配置：`configs/qwen25vl_chartqa.yaml`，里面包含模型、数据集、LoRA、训练、评估和输出目录。
- 写了配置加载函数 `load_config`，保证后续训练、评估和 Demo 都从同一份配置读参数。
- 写了 ChartQA 样本规范化逻辑，把原始数据里的 `image`、`query`、`label` 等字段统一成 `image`、`question`、`answer`、`question_type`。
- 写了 Qwen2.5-VL 的 chat messages 格式化逻辑，把一条图表问答样本转成 system、user、assistant 三轮消息。
- 写了 AutoDL 环境检查脚本，用来在远程机器上检查 Python、CUDA、GPU 和依赖安装。
- 写了本地单元测试，确认配置、数据规范化和 prompt 格式化能在不加载模型、不下载数据、不占用 GPU 的情况下跑通。

### 解决了什么问题

这一阶段解决的是“训练前不确定性”的问题。

如果直接上云端跑训练，很容易把错误混在一起：可能是配置问题、字段问题、prompt 格式问题、模型加载问题，也可能是显存问题。现在先把配置、数据字段和 prompt 这些纯代码部分在本地测掉，后面云端失败时就更容易判断问题是不是来自模型加载、依赖版本或显存。

### 成果

- 本地已经有了可测试的训练前链路。
- 后续训练脚本可以复用同一套配置、数据样本结构和 prompt 格式。
- Claude Code 远程执行时有了明确的环境检查入口。
- 项目开始具备“工程化微调项目”的样子，而不是只停留在 notebook 复现。

### 面试问题与答案

**问题 1：为什么你没有一开始就直接写训练脚本，而是先做配置、数据和 prompt 脚手架？**

回答：因为多模态微调项目里，训练失败不一定是模型问题，很多时候是数据字段、prompt 格式、配置路径或者依赖环境出了问题。我先把不需要 GPU 的部分在本地拆出来测试，能提前排除一批低级但很耗时的问题。这样到了 AutoDL 上跑 smoke training 时，如果失败，就更容易定位到模型加载、显存、量化配置或者训练参数，而不是从头猜。

**问题 2：ChartQA 原始样本为什么要先规范化成统一的 `ChartQASample`？**

回答：因为不同数据集或者参考 notebook 里的字段命名可能不一样，比如有的叫 `query` 和 `label`，有的叫 `question` 和 `answer`。如果训练、评估、Demo 每个地方都直接读原始字段，后面换数据集或扩展业务报表样例会很乱。我先定义统一结构，让下游只关心 `image`、`question`、`answer`，这样数据源变化时主要改数据适配层，不影响训练和推理逻辑。

**问题 3：Qwen2.5-VL 的 prompt 为什么要做成 messages，而不是简单拼接字符串？**

回答：因为 Qwen2.5-VL 是多模态 chat 模型，输入里既有图片也有文本问题。messages 结构可以清楚表达 system 指令、user 的 image/text 内容和 assistant 答案，后续再交给 processor 的 chat template 处理。这样更贴近模型官方使用方式，也方便训练时对 assistant 部分做 label，避免把用户问题也当作模型要学习输出的内容。

**问题 4：这个阶段为什么不下载完整 ChartQA 或加载 Qwen2.5-VL？**

回答：这是有意控制范围。完整数据集下载和 7B 多模态模型加载都依赖网络、磁盘、显存和 CUDA 环境，适合放到 AutoDL 上验证。本地阶段只验证纯逻辑，保证代码结构正确。这样本地迭代快，远程验证也更聚焦。

**问题 5：如果面试官问这个项目和普通 notebook 复现有什么区别，可以怎么说？**

回答：普通 notebook 复现往往把数据处理、prompt、模型加载、训练和评估写在一个文件里，能跑但不好维护。我这个项目把它拆成配置层、数据适配层、prompt 层、后续训练评估层，并且用测试覆盖不依赖 GPU 的部分。这样更接近真实工程项目：可以复现实验，可以替换数据，可以分析 badcase，也能把训练和 Demo 分开维护。

## 阶段 2：远程 Smoke Training 入口

### 做了什么

这一阶段继续基于 Kaggle 上的 Qwen2.5-VL + ChartQA 微调 notebook，把里面的训练关键步骤整理成可以在 AutoDL 上直接运行的项目脚本。

主要完成了这些事：

- 写了 `src/chartvqa/modeling.py`，专门负责模型相关逻辑。
- 把 4-bit BitsAndBytes 量化参数整理成配置构造函数，使用 NF4、double quantization 和 bfloat16 compute。
- 把 LoRA 参数整理成配置构造函数，第一版使用 `r=8`、`alpha=16`、`dropout=0.1`，target modules 是 `q_proj` 和 `v_proj`。
- 写了 Qwen2.5-VL model 和 processor 的加载入口，训练模式下会关闭 cache，并启用 gradient checkpointing。
- 写了 `src/chartvqa/training.py`，负责 TRL `SFTConfig` 参数、数据格式化、VLM collator 和 `SFTTrainer` 组装。
- 写了 `scripts/train_lora.py`，作为 Claude Code 在 AutoDL 上运行 smoke training 的标准命令入口。
- 给训练入口加了 `--max-steps` 参数，方便先跑 1 step 验证链路。
- 本地新增测试，只验证参数构造、数据格式化和命令行参数，不加载大模型、不占用 GPU。

### 解决了什么问题

这一阶段解决的是“云端到底运行什么命令”的问题。

如果没有标准训练入口，远程训练很容易变成在 notebook 里手动改 cell，失败后也不好复现。现在训练入口固定为：

```bash
python scripts/train_lora.py --config configs/qwen25vl_chartqa.yaml --max-steps 1 --skip-initial-eval
```

这样 Claude Code 在 AutoDL 上只需要执行固定命令，并把日志反馈回来。后面如果失败，我们能根据日志判断是模型下载、依赖版本、显存、数据集还是 collator 的问题。

### 成果

- 项目已经具备远程 1-step smoke training 的脚本入口。
- LoRA 和 4-bit 量化参数从 notebook 迁移到了工程化配置。
- 本地测试能验证训练参数链路，不需要 GPU。
- 下一步可以交给 Claude Code 在 AutoDL 上真实加载模型并跑 1 step。

### 面试问题与答案

**问题 1：为什么第一版 LoRA 只选择 `q_proj` 和 `v_proj`？**

回答：这是参考 Qwen2.5-VL + ChartQA notebook 的保守配置，也是很多 LoRA 微调里常见的低成本选择。`q_proj` 和 `v_proj` 属于 attention 里的关键投影层，调整它们可以影响模型对输入信息的关注方式，同时可训练参数量比较少，适合单卡 4090 做第一版 smoke test。等链路跑通后，再考虑扩大到更多模块或提高 rank。

**问题 2：为什么要用 4-bit QLoRA，而不是直接全量微调？**

回答：Qwen2.5-VL-7B 这种多模态 7B 模型全量微调显存成本很高，单卡 4090 24GB 不适合直接全量训练。4-bit QLoRA 通过量化冻结基础模型，再只训练 LoRA adapter，可以大幅降低显存占用，同时保留一定任务适配能力。这很适合项目第一阶段验证数据、prompt 和训练流程。

**问题 3：`--max-steps 1` 的 smoke test 有什么意义？它又不能代表最终效果。**

回答：`--max-steps 1` 不是为了看效果，而是为了验证工程链路：模型能不能下载和加载，数据能不能读，prompt 和 collator 能不能喂给 processor，LoRA 能不能注入，反向传播能不能跑，adapter 能不能保存。只有这一步成功，后面扩大 steps、数据比例和 epoch 才有意义。

**问题 4：为什么训练时要关闭 `use_cache` 并启用 gradient checkpointing？**

回答：训练时 `use_cache` 主要服务推理加速，但会占额外显存，也可能和 gradient checkpointing 不兼容。gradient checkpointing 会用重新计算换显存，适合单卡 4090 这种显存有限的环境。这个项目第一阶段更看重能稳定跑通，所以优先省显存。

**问题 5：VLM 的 collator 和普通文本模型有什么不同？**

回答：普通文本模型通常只处理 token ids 和 attention mask。VLM collator 还要处理图片输入，要把每条样本里的 chat messages 转成文本模板，同时提取图片传给 processor。最后还要构造 labels，并把 padding token mask 成 `-100`，避免模型学习 padding。这个步骤如果写错，训练可能能启动但 loss 和学习目标都是错的。

**问题 6：如果远程 smoke training 失败，你会怎么排查？**

回答：我会先看失败阶段。如果模型还没加载就失败，优先查依赖版本、模型下载、HuggingFace 访问和显存。 如果数据加载失败，查 ChartQA split 和字段。 如果进入 collator 后失败，查 messages 结构和 processor 输入。 如果反向传播 OOM，先调低 `max_pixels`、batch size、gradient accumulation 或跳过初始 eval。这样按阶段排查比只看最后一行报错更可靠。
